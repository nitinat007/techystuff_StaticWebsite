All of them use Map data structure to keep the data but one main difference is locking.

1.HashMap

  • for single threaded environment
  • Not thread-safe
  • can keep one null key and multiple null value.

  • Hashtable is an implementation of Map data structure
  • This is a legacy class in which all methods are synchronized on Hashtable instances using synchronized keyword.
  • Thread-safe as it’s method are synchronized

What is  ConcurrentHashMap? Why should you prefer using it over Hashtable? (introduced in java 1.5)

  • ConcurrentHashMap implements Map data structure and also provides thread safety like Hashtable.
  • It works by dividing the complete hashtable array into segments or portions and allowing parallel access to those segments.
  • The locking is at a much finer granularity at a hashmap bucket level.
  • Reads can happen very fast while writes are done with a lock on segment level or bucket level.
  • Use ConcurrentHashMap when you need very high concurrency in the application.
  • It is a thread-safe without synchronizing the whole map. There is no locking at the object level.
  • ConcurrentHashMap doesn’t throw a ConcurrentModificationException if one thread tries to modify it while another is iterating over it.
  • ConcurrentHashMap does not allow NULL values, so the key can not be null in ConcurrentHashMap
  • ConcurrentHashMap doesn’t throw a ConcurrentModificationException if one thread tries to modify it, while another is iterating over it. ConcurrentHashMap uses fail-safe iterator (not fail-fast iterator). But if a hashtable is structurally modified during iteration, concurrentModificationException occurs.  ConcurrentHashMap makes a copy of the internal data structure (object array) and iterates over the copied data structure.
  • Ways to initialize ConcurrentHashMap:

ConcurrentHashMap<String,String> premiumPhone = new ConcurrentHashMap<String,String>();

ConcurrentHashMap<String,String> premiumPhone  = new ConcurrentHashMap (int initialCapacity, float loadFactor, int concurrencyLevel);

Can a subclass method declare an exception if the superclass method doesn’t throw any exception?

If the super class method does not declare any exception, then the subclass overridden method cannot declare checked exceptions but it can declare unchecked exceptions.

If the Super class method throws an object of NullPointerException class, then the Subclass method can either throw the same exception, or can throw no exception, but it can never throw an object of Exception class (parent of NullPointerException class).

What is a Chained Exception in Java? 

Chained Exception was added to Java in JDK 1.4. This feature allows you to relate one exception with another exception.

getCause() and initCause() are the two methods added to Throwable class.

getCause() method returns the actual cause (lower level info) associated with the current exception.
initCause() set an underlying cause(exception) with invoking exception. eg:

import java.io.IOException;
public class ChainedException
 {
  public static void divide(int a, int b)
  {
   if(b==0)
   {
    ArithmeticException ae = new ArithmeticException("top layer");
    ae.initCause( new IOException("cause") );
    throw ae;
   }
   else
   {
    System.out.println(a/b);
   }
  }

 public static void main(String[] args)
 {
  try {
   divide(5, 0);
  }
  catch(ArithmeticException ae) {
   System.out.println( "caught : " +ae);
   System.out.println("actual cause: "+ae.getCause());
  }
 }
}

Explain InputStream, OutputStream, Reader, Writer and classes extending these Classes.

InputStream is an abstract class with overloaded read() abstract methods and is used for reading Byte Stream. OutputStream is an abstract class with overloaded write() abstract methods and is used for writing Byte Stream. Reader is an abstract class for reading Character Stream. Writer is an abstract calls for writing Character Stream.

Classes extending abstract OutputStream: FileOutputStream, ObjectOutputStream, PrintStream etc

Classes extending abstract Reader: BufferedReader,FileReader, StringReader etc

Classes extending abstract Writer: BufferedWriter,FileWriter, StringWriter, PrintWriter etc

It is recommended to use FileReader for character Stream and InputStreamReader for byte stream.

Why do we use BufferedInputStream & BufferedOutputStream  in Java? Also 

Reading and writing using byte by byte is way too expensive as it needs disk access. We use buffering to overcome this. Here we read/write a block of bytes into a memory buffer (byte array) and finally read/write (flush) from this buffer in a single IO operation. Default size of the memory buffer is 8192 byte,which can be changed by a programmer. We have BufferedInputStream & BufferedOutputStream for this.  These streams do not work independently. They work with other streams like FileInput/FileOutput streams. They have buffering as the main functionality and do not deal with reading and writing. Eg

BufferedInputStream bis = new BufferedInputStream(new FileInputStream(“go.jpg”));

What is Serialisation and Deserialization in Java?

Serialization is a process of converting an object into a sequence of bytes which can be persisted to a disk or database or can be sent through streams.  ObjectOutputStream connected to a FileOutputStream. Deserialization is the reverse process where the byte stream is used to recreate the actual Java object in memory. Deserialization does not call the constructor of the class. But if there is any non serializable ancestor, then the ancestor constructor is run.

Below are the differences between an abstract class and an Interface

  1. You can not extend more than one class because Java does not support multiple inheritances but you can implement multiple inheritance in Java.
  1.  Interface is better suited for Type declaration and abstract class is more suited for code reuse and evolution perspective.

Why was the default implementation of interface introduced in Java 8 ?

Default implementation of an interface allows us to add new methods to an interface that are automatically available in the implementations. Thus, there’s no need to modify the implementing classes. In this way, backward compatibility is neatly preserved without having to refactor the implementations. With java 8 interfaces can contain implementation of default methods and static methods. Default method must have a keyword default.

When to use an interface and when to use an abstract class?

When you know partial behavior of a class you can have a predefined method for known behavior and for declare methods whose behavior is unknown/may vary as per the implementation in an abstract method. Eg. HttpServlet is an abstract class. Methods for creation and destruction are predefined and developers have to write their own implementation for certain other methods (like service(ServletRequest req,ServletResponse res) ).

Interface is created when you know only the contract and you do not know the behavior at the time of creation. Any class that implements this interface has to write behaviour of each and every contract.  Eg: Map is interface , HashMap , HashTable , TreeMap , WeekHashMap are implementation of the interface. 

In General prefer interfaces to abstract classes. Additional capability (mixins) can be added to a class by implementing an interface. Even when a class already extends a class (or already implements another interface) we can still implement this mixin interface. This would not be possible with abstract class. 

Prefer abstract class over interface if you have to define core identity of subclass. I.e you know the abstract behavior (identity) of sub classes and state. Programmer tries to give skeletal implementation using abstract class. Eg: Animal class with variables defined (eg name, makeNoise,isWild etc) should be an abstract class rather than an interface.

What is Inner Class?

If you want to create a class which is to be used only by the enclosing class, then it is not necessary to create a separate class file for that. Instead, you can add it as “Inner Class”, i.e class within class.

If a class named Outer has a class named Inner within it, this is how you instantiate the inner class

  Outer ot = new Outer();
Outer.Inner innerObject = ot.new Inner();

Accessing methods ( Here Outer is the outer class name and Inner is the inner class name)

  1. Outer class can access inner class method by: innerObject.innerMethod()
  2. Inner class can access the outer class method by: outerMethod(). Here outerMethod of Outer class is invoked only if Inner class does not have the same method.
  3. From the inner class this.go() will always refer to the inner class’s go() method.
  4. From the inner class Outer.this.go() will always refer to the Outer class’s go() method.

What are the algorithms used in GC?

Mark and sweep Algorithm: Start marking the live (reachable) objects from GC root. In the sweep page, find and reclaim memory of all objects in the heap area that are not live. It is needed to pause the application so that there is no change in objects in the heap during the time when GC does its work. 

Similarly there is a mark-sweep-compact algorithm: This additionally compacts the space in the heap by moving all live objects to the beginning of the memory region so that objects of large memory size can be created.

Similarly there is a mark and copy algorithm: Here two different memory regions are maintained. After the mark and sweep all live objects are moved to one region. This algo is better as this marking and copying can happen parallely as soon as the object is marked.

What are the Garbage Collecting strategies in HotSpot JVM?

Oracle’s HotSpot is by far the most common JVM. All of HotSpot’s garbage collectors implement a generational garbage collection strategy that categorizes objects by age. HotSpot has four garbage collectors:

  1. Serial: All garbage collection events are conducted serially in one thread. Compaction is executed after each garbage collection. JVM argument : -XX:+UseSerialGC
  2. Parallel: Multiple threads are used for minor garbage collection. A single thread is used for major garbage collection and Old Generation compaction. Alternatively, the Parallel Old variant uses multiple threads for major garbage collection and Old Generation compaction. JVM argument: -XX:+UseParallelGC
  3. CMS (Concurrent Mark Sweep): Multiple threads are used for minor garbage collection using the same algorithm as Parallel. Major garbage collection is multi-threaded, like Parallel Old, but CMS runs concurrently alongside application processes to minimize “stop the world” events (i.e. when the garbage collector running stops the application). No compaction is performed. JVM argument: -XX:+UseConcMarkSweepGC
  4. G1 (Garbage First): The newest garbage collector is intended as a replacement for CMS. It is parallel and concurrent like CMS, but it works quite differently under the hood compared to the older garbage collectors. JVM argument : -XX:+UseG1GC

What are the different sections in the JVM heap?

young generation:

The Young Generation is where all new objects are allocated and aged. When the young generation fills up, this causes a minor garbage collection. A young generation full of dead objects is collected very quickly. Some survived objects are aged and eventually move to the old generation.

old generation:

The Old Generation is used to store long surviving objects. Typically, a threshold is set for young generation object and when that age is met, the object gets moved to the old generation. Eventually the old generation needs to be collected. This event is called a major garbage collection

permanent generation:

The Permanent generation contains metadata required by the JVM to describe the classes and methods used in the application. The permanent generation is populated by the JVM at runtime based on classes in use by the application.

PermGen has been replaced with Metaspace since Java 8 release.

PermSize & MaxPermSize parameters will be ignored now

What is method binding in java?

Method binding means binding a method to method declaration and its implementation. Compiler checks if the reference type has a compatible method. If found then it writes the method signature into the byte code it generates. If not found it throws a compiler error. This is always done by the compiler, irrespective of the methods being static or non static(instance type) . This process is called method – signature binding. If the method is static, then the compiler decides the method to be invoked (based on reference type). But if the method is of instance type (non static) then the JVM decides the method to be invoked (based on the object type) at runtime. So, for static methods the method binding happens at compile time itself (early binding) . But binding of methods of instance type has to wait till the runtime as it is dependent on object type (late binding). At runtime if the method is found at the object’s reference, the JVM invokes it else it tries to find the methods one step above in the inheritance tree. Polymorphism and method overriding follows late binding.

Instance methods are late bounded but instance fields (including static) are early bounded.

What does Method Area Contain?

Method Area contains all data pertaining to a class. This includes meta information of a class (class name, superclass name etc). Basically for every object in the heap we have class data in this method area. Before creating any object, the JVM checks in Method Area for the presence of the class data of the type and ( loads the same if not found) returns the reference of the class object. This also contains the byte code of each method, static fields info, runtime constant pool. Method area and heap are shared across all the threads.

Prior to Java 8, the method area was allocated in permanent generation space (inside JVM memory).  From Java 8 onwards Method area is moved to native heap and permanent Generation area is completely removed. What was permGen prior to java 8 is now Metaspace. In metaspace there is no memory max size unlike in permGen where there was a default max size.

How does a JIT compiler help in further speeding of execution?

JIT helps in further speeding of the execution. JVM monitors the frequency of each piece of the byte code during execution. Bytecodes which are very frequently executed are referred to as hot spots. These hot spots are then given to the JIT compilers (a component of JVM). JIT compiler converts these hotspots into native machine code. This machine code is then cached (CodeCache present in native Area of memory). In future whenever code in hot spots is to be run, instead of interpreting the byte code by interpreter, the corresponding cached machine code is directly run. This results in fast execution. This is also sometimes referred to as dynamic compilation.

What are the steps involved in Class Loading?

Classes found via less trustworthy Class Loader go through high level of security checks by JVM.

Verification Step:  This is done by the bytecode verifier which verifies if the  bytecode is not tampered/malformed. This also does semantics checks (eg no illegal method overloading, final classes are not sub-classed etc)

Preparation Step: Allocated memory for static fields and initialized with default value. Throws OutOfMemoryError if no space is left. If a class is loaded because an instance of it is created then memory is allocated for this instance object as well (done only after all static fields are initialized). All the static fields across the class hierarchy are initialized before any instance fields are initialized.  

Resolution Step: This is an optional step. This step resolves and loads the classes/interfaces that are referenced in the class. These references are stored in the Constant pool (this also has compile-time constants and string literals. Constant pools can be seen in .class file when disassembled using disassembler). Resolution also ensures that class has permissions to access them. At runtime, these references are replaced by actual references in the memory. This is called dynamic linking. Dynamic linking can happen in two ways

  1. Eager loading: This happens just after the preparation step. All referenced classes are loaded before current class is loaded.
  2. Lazy loading: This happens after the initialization step. This is on demand which means that this happens when a statement that includes the references are actually executed. JVM implements lazy loading.

Explain the Class Loading process

First rt.jar (rt means runtime which includes core java library) present in jdk/jre/lib is opened and all classes (like object, Serializable, String etc) are loaded by bootstrap class loader.  After this the actual class (inside com.<package>.jvm ) which contains the main method is loaded. Static initializer block in this class will run first. Instance initializer block at this point is not initialized yet as instance has not been created yet. Next the main (static) method is invoked. Inside the main method, some other class might be loaded. If there is a statement accessing the static field of another class, then also another class is not loaded as the static field is compile time constant. If there is a statement accessing a non static field of another class then the interface, super class and this class (containing the field) is loaded in the same order.  Classes/interfaces might be loaded without actually initializing. You know that an  interface is loaded either if one of the fields which is not a compile time constant is used or a static method is invoked. Now if these classes/methods use another methods or fields from another class then lazy loading of these classes happen. Before a constructor of any class is called its instance initializer block will be called. Note that accessing compile time constants from a class will not load the class as its value known from compile time itself. 

You can add an argument -verbose:class to the program to see the classes loaded when a program starts running.

How does a java program run ?

Let’s say we have a program called ‘Hello.class’. When we run the command ‘java Hello’, an instance of JVM is created and loaded into memory. JVM then in turn loads ‘Hello.class’ program into memory and executes it. One JVM instance runs only one java application.

When JVM is created, a chunk of memory (called runtime data areas or JVM memory) is obtained from the underlying operating system. In order to load the ‘Hello.class’ , the JVM invokes a component called class loader. The class loader’s responsibility is to locate the Hello.class and load the corresponding byte code into memory. JVM Memory has two important areas: heap and method area. Byte code once loaded is verified by a component called the bytecode verifier. Bytecode verifier verifies whether the loaded byte code is not corrupted or tampered by hackers. Once bytecode verification passes, the bytecode is passed to the execution engine (which includes JIT compiler and interpreter). This interpreter helps in achieving platform independence and JIT compiler help in very fast execution speed. Along with this, we have a component called Garbage collector (GC) which has the responsibility of managing the memory of the JVM. GC reclaims the memory no longer in use. There is a component called security manager, which ensures security, one of the core responsibilities of JVM. The security manager can help in running code in a more restricted environment (like preventing code to access file systems) called Sandbox.

Whenever a new class instance is created, memory space is allocated for it with room for all the instance variables declared in the class type and all the instance variables declared in each superclass of the class type, including all the instance variables that may be hidden. If there is not sufficient space available to allocate memory for the object, then the creation of the class instance completes abruptly with an OutOfMemoryError (does not throw null). What is the difference between Virtual Machine and Sandboxed Environment?

A virtual machine provides a simulated computer that guest applications will see as the whole system. It is not necessarily sandboxed since it can provide write access to any shared location on the host computer.

A sandboxed environment is not necessarily virtualized: it does not have to simulate a computer, just restrict access permissions for running applications. Javascript within your browser provides such an environment without virtualization.
