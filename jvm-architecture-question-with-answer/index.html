What does Method Area Contain?

Method Area contains all data pertaining to a class. This includes meta information of a class (class name, superclass name etc). Basically for every object in the heap we have class data in this method area. Before creating any object, the JVM checks in Method Area for the presence of the class data of the type and ( loads the same if not found) returns the reference of the class object. This also contains the byte code of each method, static fields info, runtime constant pool. Method area and heap are shared across all the threads.

Prior to Java 8, the method area was allocated in permanent generation space (inside JVM memory).  From Java 8 onwards Method area is moved to native heap and permanent Generation area is completely removed. What was permGen prior to java 8 is now Metaspace. In metaspace there is no memory max size unlike in permGen where there was a default max size.

What does JVM stack ,PC and native Stack contain ?

JVM stack contains order of invocation of methods, state and info of local variables in each method. 

Native Stack contains methods invoked in the native library (of C, C++) as JVM heavily relies on them (say for IO)

PC is similar to instructor pointer in the native CPUs. This contains the memory address of the next instruction. It is just the pointer to the method (method byte code) in the method area. 

JVM Stack, PC, and native Stack are created per thread.

How does a JIT compiler help in further speeding of execution?

JIT helps in further speeding of the execution. JVM monitors the frequency of each piece of the byte code during execution. Bytecodes which are very frequently executed are referred to as hot spots. These hot spots are then given to the JIT compilers (a component of JVM). JIT compiler converts these hotspots into native machine code. This machine code is then cached (CodeCache present in native Area of memory). In future whenever code in hot spots is to be run, instead of interpreting the byte code by interpreter, the corresponding cached machine code is directly run. This results in fast execution. This is also sometimes referred to as dynamic compilation.

What is CodeCache in JVM? What happens when this cache gets full? 

JVM JIT generates compiled code and stores that in a memory area called CodeCache. The default maximum size of the CodeCache on most of the platforms is 48M. If any application needs to compile a large number of methods resulting in a huge amount of compiled code then this CodeCache may become full. In such a case the compiler is disabled. Any further compilations are stopped, and a message like this gets logged “CodeCache is full. Compiler has been disabled.” You can increase the size by passing the JVM argument “-XX:ReservedCodeCacheSize=n.”

When is class and interface loaded in java?

First time a class is loaded if 
  1. New instance is created
  2. Invoke static method
  3. Accessing static field (except compile time constants as values are known at compile time itself)
  4. A subtype ( here subclass) is loaded
  5. Run from command line
  6. Reflection

First time an interface is loaded also for all the above reasons, except 1st point.

What are class loaders in java? 

Class loaders are responsible for loading Java classes during runtime dynamically to the JVM. They are part of the JRE . They are responsible for loading classes into the memory. There are three types of class loader : Application, Extension, Bootstrap. Eg

Class loader of this class:sun.misc.Launcher$AppClassLoader@18b4aac2

Class loader of Logging:sun.misc.Launcher$ExtClassLoader@3caeaf62

Class loader of ArrayList:null 

Bootstrap class loader is written in native language, not java. Bootstrap class loader serves as a parent of all the other ClassLoader instances.The java.lang.ClassLoader.loadClass() method is responsible for loading the class definition into runtime. It tries to load the class based on a fully qualified name whenever the JVM requests for the class. If this parent class loader doesn’t find the class, then the child class will call java.net.URLClassLoader.findClass() method to look for classes in the file system itself. If the last child class loader isn’t able to load the class either, it throws java.lang.NoClassDefFoundError or java.lang.ClassNotFoundException.

What is class Object?

Class object contains the meta information of the class. JVM creates an object of a class with the class object information. To get information about the class these methods are useful. like getName(), getSuperClass(), isInterface(), getInterfaces(), getClassLoader() → Class Loader that loaded this class  etc. 

Class, Interface, primitive, void, array all of them have class object.

What are the steps involved in Class Loading?

Classes found via less trustworthy Class Loader go through high level of security checks by JVM.

Verification Step:  This is done by the bytecode verifier which verifies if the  bytecode is not tampered/malformed. This also does semantics checks (eg no illegal method overloading, final classes are not sub-classed etc)

Preparation Step: Allocated memory for static fields and initialized with default value. Throws OutOfMemoryError if no space is left. If a class is loaded because an instance of it is created then memory is allocated for this instance object as well (done only after all static fields are initialized). All the static fields across the class hierarchy are initialized before any instance fields are initialized.  

Resolution Step: This is an optional step. This step resolves and loads the classes/interfaces that are referenced in the class. These references are stored in the Constant pool (this also has compile-time constants and string literals. Constant pools can be seen in .class file when disassembled using disassembler). Resolution also ensures that class has permissions to access them. At runtime, these references are replaced by actual references in the memory. This is called dynamic linking. Dynamic linking can happen in two ways

  1. Eager loading: This happens just after the preparation step. All referenced classes are loaded before current class is loaded.
  2. Lazy loading: This happens after the initialization step. This is on demand which means that this happens when a statement that includes the references are actually executed. JVM implements lazy loading.

What is initialization in the lifetime of a type?

Initialization step initializes static fields with user defined initial values (as previous step had assigned a default values). Also note that an interface is initialized only if one of the static methods is accessed or one of the non-compiled time fields is accessed (via a constructor or method).

Explain the Class Loading process

First rt.jar (rt means runtime which includes core java library) present in jdk/jre/lib is opened and all classes (like object, Serializable, String etc) are loaded by bootstrap class loader.  After this the actual class (inside com..jvm ) which contains the main method is loaded. Static initializer block in this class will run first. Instance initializer block at this point is not initialized yet as instance has not been created yet. Next the main (static) method is invoked. Inside the main method, some other class might be loaded. If there is a statement accessing the static field of another class, then also another class is not loaded as the static field is compile time constant. If there is a statement accessing a non static field of another class then the interface, super class and this class (containing the field) is loaded in the same order.  Classes/interfaces might be loaded without actually initializing. You know that an  interface is loaded either if one of the fields which is not a compile time constant is used or a static method is invoked. Now if these classes/methods use another methods or fields from another class then lazy loading of these classes happen. Before a constructor of any class is called its instance initializer block will be called. Note that accessing compile time constants from a class will not load the class as its value known from compile time itself. 

You can add an argument -verbose:class to the program to see the classes loaded when a program starts running.
